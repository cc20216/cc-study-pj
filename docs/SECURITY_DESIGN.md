# 智能学习管理系统 - 安全设计

## 1. 认证授权机制

### 1.1 用户登录实现
- **登录流程**：
  1. 用户输入用户名和密码
  2. 系统使用Django的`authenticate()`函数验证用户凭据
  3. 验证成功后，使用`login()`函数创建会话
  4. 会话信息存储在数据库中，通过sessionid cookie关联
  5. 登录成功后重定向到首页

- **安全措施**：
  - 密码使用Django内置的PBKDF2算法哈希存储
  - 登录失败时不显示具体错误原因（避免暴力破解）
  - 登录页面使用CSRF保护

### 1.2 权限控制实现
- **基于装饰器的权限控制**：
  - 使用`@login_required`装饰器保护需要登录的视图
  - 示例：`@login_required`装饰的`study_session`视图

- **用户数据隔离**：
  - 所有查询都使用`user=request.user`进行过滤
  - 确保用户只能访问自己的数据
  - 示例：`TodoTask.objects.filter(user=user)`

- **模板访问控制**：
  - 用户只能访问示例模板或自己创建的模板
  - 使用Q对象实现：`Q(is_example=True) | Q(creator=user)`

### 1.3 会话管理
- **会话过期**：
  - Django默认会话过期时间为2周
  - 可在settings.py中配置`SESSION_COOKIE_AGE`

- **会话验证**：
  - 每次请求都会验证sessionid的有效性
  - 会话数据存储在数据库中，确保安全性

## 2. 数据隔离策略

### 2.1 防止用户越权访问的技术措施
- **查询过滤**：
  - 所有数据库查询都添加用户过滤条件
  - 示例：`DailyProgress.objects.filter(user=user)`

- **模板权限检查**：
  - 在提交进度时检查用户是否有权访问该模板
  - 代码：
    ```python
    if not (template.is_example or template.creator == user):
        messages.error(request, '您无权访问该模板')
        return redirect('daily_progress')
    ```

- **URL参数验证**：
  - 验证URL中的模板ID是否属于当前用户
  - 示例：`StudyTemplate.objects.filter(id=template_id, creator=user)`

### 2.2 数据访问控制
- **管理员权限**：
  - 只有超级用户可以访问Django admin
  - 普通用户无法访问管理后台

- **文件上传安全**：
  - 头像和玩偶图片上传使用`ImageField`
  - 上传路径限制在指定目录
  - 文件名使用随机生成，避免路径遍历攻击

### 2.3 数据传输安全
- **HTTPS**：
  - 生产环境建议使用HTTPS
  - 可在Nginx配置中启用SSL

- **CSRF保护**：
  - Django默认启用CSRF保护
  - 所有POST请求都需要包含CSRF token
  - 模板中使用`{% csrf_token %}`标签

## 3. 安全测试用例

### 3.1 XSS测试
- **测试用例1**：输入包含JavaScript的任务内容
  - **输入**：`<script>alert('XSS')</script>`
  - **预期**：系统应过滤或转义该输入
  - **实际**：Django模板自动转义，显示为纯文本

- **测试用例2**：输入包含HTML的学习资料内容
  - **输入**：`<h1>测试</h1><script>alert('XSS')</script>`
  - **预期**：系统应过滤或转义该输入
  - **实际**：Django模板自动转义，显示为纯文本

### 3.2 CSRF测试
- **测试用例1**：跨站提交表单
  - **操作**：使用其他网站的表单提交到本系统的登录接口
  - **预期**：提交失败，CSRF验证失败
  - **实际**：Django CSRF保护机制阻止提交

- **测试用例2**：修改CSRF token
  - **操作**：手动修改表单中的CSRF token
  - **预期**：提交失败，CSRF验证失败
  - **实际**：Django CSRF保护机制阻止提交

### 3.3 SQL注入测试
- **测试用例1**：在用户名输入中注入SQL
  - **输入**：`admin' --`
  - **预期**：登录失败，系统安全处理
  - **实际**：Django ORM使用参数化查询，防止SQL注入

- **测试用例2**：在URL参数中注入SQL
  - **输入**：`/template/edit/1' OR 1=1 --`
  - **预期**：系统返回404或错误页面
  - **实际**：Django URL路由系统安全处理，返回404

### 3.4 权限测试
- **测试用例1**：访问其他用户的模板
  - **操作**：尝试访问ID为999的模板（假设不属于当前用户）
  - **预期**：系统拒绝访问
  - **实际**：系统返回错误消息，重定向到安全页面

- **测试用例2**：修改其他用户的任务
  - **操作**：尝试删除ID为999的任务（假设不属于当前用户）
  - **预期**：系统拒绝操作
  - **实际**：系统返回404或错误消息

### 3.5 敏感信息泄露测试
- **测试用例1**：查看错误页面
  - **操作**：触发一个错误，查看错误页面
  - **预期**：生产环境中不显示详细错误信息
  - **实际**：DEBUG=False时显示友好错误页面

- **测试用例2**：查看页面源代码
  - **操作**：查看登录页面源代码
  - **预期**：不包含敏感信息
  - **实际**：页面源代码不包含敏感信息

### 3.6 密码强度测试
- **测试用例1**：使用弱密码注册
  - **输入**：密码长度小于8位
  - **预期**：系统拒绝注册，提示密码强度不足
  - **实际**：注册视图验证密码长度，拒绝弱密码

- **测试用例2**：使用常见密码
  - **输入**：密码为"12345678"
  - **预期**：系统允许注册（当前实现）
  - **实际**：系统允许注册（建议增加密码强度检查）

### 3.7 文件上传测试
- **测试用例1**：上传非图片文件
  - **操作**：尝试上传.txt文件作为头像
  - **预期**：系统拒绝上传
  - **实际**：ImageField验证文件类型，拒绝非图片文件

- **测试用例2**：上传恶意图片文件
  - **操作**：尝试上传包含恶意代码的图片文件
  - **预期**：系统上传文件但不会执行恶意代码
  - **实际**：文件存储在服务器但不会被执行

## 4. 安全配置建议

### 4.1 生产环境配置
- **DEBUG模式**：设置为False
- **SECRET_KEY**：使用随机生成的复杂字符串
- **ALLOWED_HOSTS**：设置为实际的域名
- **CSRF_COOKIE_SECURE**：设置为True（使用HTTPS时）
- **SESSION_COOKIE_SECURE**：设置为True（使用HTTPS时）

### 4.2 数据库安全
- **数据库用户权限**：使用最小权限原则
- **密码存储**：使用Django内置的密码哈希
- **数据备份**：定期备份数据库

### 4.3 服务器安全
- **防火墙**：配置防火墙规则，只开放必要端口
- **定期更新**：定期更新系统和依赖包
- **日志监控**：监控系统日志，及时发现异常

### 4.4 代码安全
- **代码审查**：定期进行代码审查
- **依赖检查**：定期检查依赖包的安全漏洞
- **安全扫描**：使用安全扫描工具检查代码

## 5. 安全事件响应

### 5.1 事件分类
- **低风险**：轻微的安全问题，如输入验证不足
- **中风险**：可能导致数据泄露的问题
- **高风险**：严重的安全漏洞，如SQL注入

### 5.2 响应流程
1. **发现**：通过监控或用户报告发现安全问题
2. **评估**：评估问题的严重程度和影响范围
3. **修复**：开发并部署修复方案
4. **验证**：验证修复是否有效
5. **记录**：记录安全事件和修复过程

### 5.3 应急响应
- **联系人**：指定安全事件联系人
- **响应时间**：定义不同风险级别的响应时间
- **沟通计划**：制定安全事件的沟通计划
